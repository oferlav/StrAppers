System Architecture: The Social Impact Engine
1. Taxonomy and Discovery Logic
The system is anchored by the United Nations Sustainable Development Goals (SDGs). Each project is not merely a "task" but is indexed within a global impact framework (e.g., Quality Education, Zero Hunger).
•	Logical Hierarchy: Projects are parented by an SDG category. This allows the system to surface initiatives based on a user's ethical alignment rather than just their technical stack.
•	The "Selection Cap": The system enforces a "Select up to 3" constraint. This is a logical filter designed to prevent "application fatigue" and ensure that users are high-intent. It forces the user to perform a value-based trade-off before engaging.
2. The Urgency Heuristic (The "Hot" Tag)
The Hot project designation is driven by a backend algorithm. It is not a manual toggle but a dynamic status triggered by three data points:
•	Role Desperation: A critical role (like "Product Manager") remains unfilled while other slots are full.
•	Temporal Pressure: The project has an upcoming milestone or "sprint start" date.
•	Engagement Velocity: A high ratio of "views" to "applicants," suggesting the project is interesting but perhaps lacks a final piece of technical clarity to close the deal.
3. Social Proof and Team-Building Logic
The Applicants sidebar serves as a transparency layer. By integrating professional profiles (LinkedIn logic), the system creates "Social Proof." A potential applicant can see if the current team is balanced.
•	Gap Analysis: If a user sees three "Frontend developers" and zero "Backend," the system logic implies a high "Need Score" for their specific skill set.
•	Relational Logic: Displaying existing members reduces the "cold start" problem for volunteers, making the project feel like a moving vehicle they can jump onto rather than a stagnant idea.

User Story: The Specialist Seeking Purpose
As a Senior Product Manager who feels disconnected from the social impact of my daily work, I want to browse projects based on their global mission and see the current professional composition of the teams, so that I can contribute my expertise to a group that is already formed but specifically lacks my strategic leadership.
The Persona's Journey
•	Discovery: The user filters by "Decent Work and Economic Growth." They ignore the general listings and look specifically for "Hot" projects, assuming these are the closest to launching.
•	Analysis: They click on SoundSense. They read the brief, which explains the technical goal (Digital Auditory Rehabilitation).
•	Validation: The user looks at the applicant list. They see a Frontend dev, a Designer, and a Backend dev. They realize the team is "engineering-heavy" but lacks a "Product Manager" to guide the MVP.
•	Action: Seeing this specific void provides the user with a sense of "Unique Value Proposition." They click Apply Now, knowing their involvement is the "tipping point" for the project's success.
Acceptance Criteria for the System
•	Constraint Check: The system must block a fourth selection and prompt the user to deselect one project to continue.
•	Data Integrity: Clicking a team member's name must trigger a secure redirect to their external professional profile.
•	Dynamic Scaling: As more applicants join, the project's "Hot" status should automatically degrade once the core roles (as defined in the project brief) are satisfied.
