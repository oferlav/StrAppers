# Option A3: Roles Table Consolidation ‚Äî Implementation Plan

## What We're Doing

Add `InstituteId` (**non-nullable** FK, default `1`), `SquadId` (nullable FK), and `IsTechnical` (bool) to the `Roles` table so that all role variants live in one table.

### The rule after A3:
| InstituteId | SquadId | Meaning |
|---|---|---|
| `null` | `null` | Global / B2C role (never modified ‚Äî the platform catalog) |
| `X` | `null` | Institute base role (replaces `InstituteRole`) |
| `X` | `Y` | Squad-scoped role (replaces `InstituteSquadRole`) |

`InstituteId` is **nullable**. `null` = global/B2C role. Any non-null value = institute-specific. InstituteId=1 is a normal institute that can have its own roles like any other. Requires an `Institutes` row with `Id = 1` (the platform institute) for B2C student routing.

`StudentRole.RoleId` continues pointing to `Roles.Id` ‚Äî no change. Institute students now get a real FK to a row with their data, no name-matching needed.

---

## Biggest Risk ‚öÝÔ∏è

**`StudentRegistration.jsx:31` and `FigmaViewerModal.jsx:19` both hardcode `roleId === 4`.**

After A3 institute UI/UX students have a new `Role.Id` (not 4). Both checks silently fail ‚Äî no error, no log. The student registers without Figma OAuth and can't edit their Figma URL in the board room.

**Fix:** Use `Role.SkillId === 3` (Figma skill; Skills table: 1=GitHub, 2=Trello, 3=Figma, 4=CRM). `StudentRegistration.jsx` already has `skillId` in the roles API response. `FigmaViewerModal` needs one field (`RoleSkillId`) added to `GetStudentByEmail` ‚Äî the data is already loaded via `.ThenInclude(sr => sr.Role)`, just not projected.

**Secondary risk:** Missing a `WHERE InstituteId = 1` filter on any of the ~8 global-role query sites causes institute roles to bleed into every role dropdown seen by all users and employers.

---

## RolesConfig Screen Invariant

**A new institute admin opens the Roles Config screen:**
1. `GET /api/Roles/use/institute/{X}` returns empty (no institute roles yet)
2. The frontend `available` panel falls back to `fetchAllRoles()` ‚Üí global roles (`InstituteId = 1`) ‚úì
3. Admin picks roles, customises CE/IsTechnical/SkillId, and saves

**On save ‚Äî the safety rule:**
- Backend creates a **new** `Role` row with `InstituteId = X`
- The original global row (`InstituteId = null`) is **never modified**

This is already enforced today: in the squad path (`RolesController.cs:294`) the backend only treats a DTO's `Id` as "update existing" when it matches an `existingSquadIds` set (IDs of existing `InstituteSquadRole` rows for that squad). A global role's ID won't be there ‚Üí always creates new. After A3 the equivalent check must be: **never update a `Roles` row where `InstituteId = 1`** ‚Äî guard this explicitly in the save logic.

## Testing Strategy

**Test-first** for every critical safety invariant ‚Äî write the test, watch it fail (proves it catches the right thing), then write the code to make it pass. Applies to:
- Phase 3: global filter leakage (institute roles must not appear in global endpoints)
- Phase 4: SaveInstituteRoles safety guard (InstituteId=1 rows must never be touched)
- Phase 5: GetRolesByCoupon returns correct institute Role.Ids

**Alongside** for mechanical plumbing changes where the logic is obvious (nav property type change, query rewrites with clear intent).

**After every phase**: run the full existing test suite to catch regressions before moving to the next phase.

**Frontend (Phase 8)**: no test framework in place ‚Äî these are manual checklist items, called out explicitly when reached.

**Script timing** (user runs these, not Claude):
- `migrate_roles_a3.sql` ‚Üí run when Phase 3 code is ready to deploy
- `migrate_roles_a3_squads.sql` ‚Üí run after Phase 6 is verified, before Phase 7 deploys

Write the tests for a phase first. Do not mark a phase complete until its tests pass.
